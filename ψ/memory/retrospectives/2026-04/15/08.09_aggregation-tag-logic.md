## Honest Feedback

**1. Rule table keyword list ไม่ได้ validate กับ data จริง**
merchant keywords ที่ draft ใน path_a_rule_table.md เช่น `mcdonald`, `grabfood`, `gong_cha` เป็น assumption จากความรู้ทั่วไป ไม่ได้ดูจาก actual merchant_name ในฐานข้อมูลของทีม ถ้า format จริงต่างออกไป เช่น `mcdonald_th`, `grab_food_delivery` เป็นต้น จะ match ไม่ตรง ควรแนะนำให้ทีม run `SELECT DISTINCT merchant_name` per MCC ก่อน map keyword

**2. Threshold 0.3 ยังไม่มีฐานจากข้อมูลจริง**
global threshold 0.3 เลือกเพราะ practical และ Tee ยอมรับ แต่ไม่ได้มาจากการดู score distribution จริง ควรระบุใน spec ชัดกว่านี้ว่า "threshold นี้ต้อง calibrate หลัง pilot run" และแนะนำ metric สำหรับ calibration เช่น precision/recall ของ tag assignment กับ ground truth sample

**3. Complex tags ใน Phase 1 list ควร flag ก่อนหน้านี้**
tags อย่าง `last_minute_booker` และ `empty_nester` อยู่ใน 117 tags แต่ implementation ซับซ้อนกว่ามาก (ต้องการ temporal comparison) ควร flag ตั้งแต่ต้น session ไม่ใช่รอ defer ไป Phase 2 ตอนท้าย เพราะถ้าทีม engineer รับ rule table ไปโดยไม่เห็น warning อาจ underestimate effort

## Lessons Learned

1. **Mapping table ที่มีอยู่ไม่ใช่ mapping table ที่ต้องการ** — ต้อง probe ว่า existing artifact ครอบ use case จริงไหม ก่อน assume ว่า "มีแล้ว"
2. **Binary tag = categorical feature** — framing นี้ทำให้ downstream usage ชัดขึ้นทันที ทั้ง scoring logic และ ML feature preparation
3. **Sub-type classification ช่วย implementation มาก** — การแบ่ง A1–A7 ทำให้ engineer รู้ว่า query pattern ไหนใช้กับ tag แบบไหน แทนที่จะเขียน logic ใหม่ทุก tag
