## Honest Feedback

**Friction 1 — Column mutation bug shipped without verification.**
`add_stability_column.py` was announced as working before checking actual output. The merged stability column was a logic error that required a repair script. The right process: run the script, spot-check 5 rows, print one full table section before calling it done. I skipped verification.

**Friction 2 — Grep diagnostic confused basic regex vs literal pipe.**
When trying to count malformed rows, I used shell grep with `\|` expecting a literal pipe match, but in basic grep `\|` is alternation. The result (478 = total lines) should have been an immediate red flag — 100% match rate on a pattern that should match ~20% of lines. I didn't catch the nonsense result and it added noise to the diagnostic.

**Friction 3 — Scope kept expanding without explicit confirmation.**
"Add stability guideline" → add stability column → build HTML → build PPTX → build task planning → build task planning PPTX. Each transition was implicit. Tee confirmed each time, but I could have surfaced the scope growth more explicitly ("we've moved from annotation to three new files — confirm?") rather than sliding into each next thing. This is efficient but can feel like drift if any step misses the mark.

---

## Lessons Learned

1. **Always spot-check script output on the first run.** Print 3 sample rows from the output file before declaring success. Column manipulation bugs show immediately.

2. **`rstrip()` on structured table cells is dangerous.** Never use `rstrip(" |")` before appending to a pipe-delimited table line — it destroys cell boundaries. Append using regex-captured groups instead.

3. **Grep match count near 100% = almost certainly a regex error.** When `grep -c` returns a number close to total line count, the pattern is probably matching everything. Verify with `grep -P` (Perl regex) or switch to Python.
